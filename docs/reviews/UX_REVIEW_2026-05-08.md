# neurospatial UX & Interface Consistency Review

**Date:** 2026-05-08
**Repo state:** `main` @ commit `e6cc25c` (review file untracked in verification checkout)
**Package version:** `0.3.0`
**Audience for this review:** maintainers
**Audience for the package:** neuroscientists (not programmers)

This is a wide-coverage audit of the `neurospatial` repository. It is intentionally **not filtered for severity or confidence** — the brief was to surface every issue, even uncertain or low-severity ones, so a downstream pass can rank them.

Findings were collected by five independent passes:

1. **Public-API consistency** (signatures, naming, return types)
2. **Errors, validation, and silent failures**
3. **User-facing documentation** (README, CLAUDE.md, docs/)
4. **Examples and tutorials**
5. **`Environment` class deep-dive** (factories, methods, mixins, lifecycle, mutability)

A short list of high-severity findings was spot-verified against the source after the audit (see "Verified" tags below).

Severity scale: **HIGH** = blocks user, silently wrong, or breaks documented workflow. **MED** = friction / cognitive load / inconsistency that wastes neuroscientist time. **LOW** = polish.
Confidence scale: **certain** / **high** / **med** / **low**.

---

## 0. Executive summary

For a package whose target user is a neuroscientist (not a programmer), the largest UX failures fall into four categories:

1. **Onboarding is broken at the front door.** The README quickstart, the `.claude/QUICKSTART.md` (which is the path the AI assistant points humans to), `docs/getting-started/quickstart.md`, and even the top-level package docstring all contain at least one immediately-failing or misleading example each. A user who copy-pastes hits a `TypeError`, `ValueError`, `ImportError`, or `AttributeError` on early snippets. The README has a simulated spike-times-to-place-field example later, but the first "Common Use Cases" path still stops at occupancy and does not foreground the canonical user-owned-spikes workflow.

2. **Pervasive parameter-name drift.** The same concept is named four different ways in different files: distance metric (`metric` / `distance_metric` / `distance_type` / `use_geodesic`), smoothing bandwidth (`bandwidth` / `smoothing_sigma` / `kernel_bandwidth`), velocity threshold (`min_speed` / `velocity_threshold` / `speed_threshold` / `threshold`). This is the single most pervasive consistency issue.

3. **Silent failure modes in numerically-sensitive code.** `detect_place_fields` silently returns `[]` for a fast-firing cell. Batch grid/border score functions silently substitute `nan` for failing neurons with no count or summary. `bin_at` and `map_points_to_bins` disagree on points outside the environment, and they are used by overlapping methods (`bin_sequence` vs `occupancy`/`interpolate`). Per CLAUDE.md's own "scientific-code-change-audit" principle, these are exactly the kinds of plausible-looking-but-wrong outputs to avoid.

4. **Polymorphic result-class shapes.** `EgocentricRateResult.ego_env` (not `env`); `ViewRateResult.view_occupancy` (not `occupancy`); `PeriEventResult.firing_rate()` is a method, while `SpatialRateResult.firing_rate` is an attribute. Code that wants to write a generic helper over result classes can't.

The rest of the document enumerates these and ~150 other findings with file:line references.

### 1.5 README canonical place-field path is not front-door enough (MED, high)

The README does include a simulated place-cell example that calls `compute_spatial_rate` (`README.md:363-369`). However, the first "Common Use Cases #1" path stops at `env.occupancy(...)`, and the most direct "I have spike times + tracking; make a rate map" workflow is not presented as the primary first-run path. A 30-second test ("can a neuroscientist with their own arrays figure out where to start?") still has unnecessary friction.

`examples/19_real_data_bandit_task.py` requires ~70 MB of `.pkl` files in `data/`, which is gitignored. `examples/README.md:31-177` table of contents stops at "08. Complete Workflow". Notebooks 09-22 are invisible to a user who reads the index. Worse, the README mis-names notebook 08 as `08_complete_workflow.ipynb`; the actual file is `08_spike_field_basics.ipynb`. Six prerequisites links would 404.

`src/neurospatial/io/nwb/` is the canonical input format for the target audience. `grep` of `examples/*.py` returns zero usage. CLAUDE.md/ADVANCED.md advertise it as a major feature.

`compute_directional_rate`, `is_head_direction_cell`, `rayleigh_test`, `plot_head_direction_tuning` are exported. Zero notebooks use them. The "big four" tuning types (place / grid / boundary / HD) are unevenly represented (place ✓, boundary ✓, HD ✗, grid ✗).

`compute_egocentric_rate`, `is_object_vector_cell`, `object_vector_score` exported. Zero notebooks use them. CLAUDE.md "Most Common Patterns #8" demonstrates them; users following the doc have nowhere to go.

No `00_start_here.md`, no task-oriented index ("Goal → notebook" map). The numeric sequence implies progression but mixes infrastructure (01-07: pure spatial discretization, ~7 hours per the README) with the actual neural analyses (08-22).

`15_simulation_workflows.ipynb` and `20_bayesian_decoding.ipynb` show 0/21 and 0/24 cells with outputs. On GitHub these render as blank pages. Two pivotal topics are invisible to passive readers.

### 1.19 Notebook prerequisites are not linked from the notebook itself (LOW, certain)

A user who lands on `11_place_field_analysis.ipynb` from a search engine has no link back to "you should read 08 first."

### 1.20 Cross-notebook references are bold text, not hyperlinks (LOW, certain)

Many notebooks say "see **02_layout_engines.ipynb**" without a relative-MD link. GitHub renders bold but doesn't navigate.

## 8. Coordinate convention issues

### 8.1 `GridProperties.peak_coords` uses `(row_offset, col_offset)` while everything else is `(x, y)` (HIGH, certain)

`src/neurospatial/encoding/grid.py:114-117`. Plotting `peak_coords[:,0]` against `bin_centers[:,0]` will silently invert axes.

`src/neurospatial/animation/transforms.py:262, 319`, `animation/backends/napari_backend.py:828`. `VideoCalibration` is public. Documented but a known footgun.

### 8.3 `is_1d` does NOT mean `n_dims == 1` (MED, certain)

`Environment.from_graph` produces an env with `is_1d=True`, `n_dims=2`. The `is_1d` is "linearized track in a 2D world". Users will assume the property names are equivalent. Rename to `is_linearized_track`.

### 8.4 `from_polar_egocentric` envs are indistinguishable from Cartesian 2D (HIGH, high)

`bin_centers[:, 0]` is distance, `[:, 1]` is angle in radians. `n_dims=2`. No `is_polar` flag. Methods that compute Euclidean distance on `bin_centers` (notably `distance_to`, `bin_at` for points specified in (x,y), the default `plot_field` axes labelled "X Position") silently produce nonsense.

### 8.5 Egocentric heading convention conflicts in adjacent docstrings (HIGH, high)

CLAUDE.md mandates egocentric `0=ahead, π/2=left, -π/2=right`. But:

- `compute_egocentric_rate` (`encoding/egocentric.py:971`) docstring says headings are `0=East allocentric` (because the function transforms allocentric heading to egocentric internally).
- `compute_view_rate` (`encoding/view.py:909`) — `0=East`.
- `compute_directional_rate` (`encoding/directional.py:1338`) — silent on what `heading=0` means.
- `is_head_direction_cell` (`encoding/directional.py:1804`) — silent.

Convention drift in user-facing functions is the kind of silent error that produces wrong figures with no test failure.

### 8.6 `min_speed` units undocumented (MED, certain)

`heading_from_velocity(positions, dt, min_speed)` (`ops/egocentric.py:600-602`). CLAUDE.md says "cm/s", QUICKSTART says "cm/s", but the function's own docstring is silent on units.

### 8.7 Distance returned from `geodesic_distance_matrix`/`distance_field` units undocumented (MED, certain)

`ops/distance.py:51, 217`. Depends on edge `weight` attribute. Reader has no signal.

### 8.8 `Environment.units` is a free-form string with zero validation (MED, certain)

`env.units = "cm"` is decorative. Nothing checks `bin_size`, `bandwidth`, `view_distance`, `min_speed`, etc. are in those units. Operations between mismatched-units envs (e.g., `composite.py`) are not blocked. `frame: str | None` has the same issue.

### 8.9 `simulate_trajectory_ou(speed_units=None)` allows mismatched units (HIGH, certain)

`src/neurospatial/simulation/trajectory.py:70`. Default speeds 0.04 / 0.08 (m/s) but bin_sizes elsewhere are typically cm. A user assuming "speed_mean=0.08 in env units" introduces 100× errors silently.

## 9. `Environment` class deep findings

### 9.1 `subset()` envs cannot be reloaded from disk (HIGH, certain)

`Environment.subset()` (`environment/transforms.py:577-668`) creates a `SubsetLayout` with `_layout_type_tag="subset"`. `to_file` succeeds. `from_file` fails: `ValueError: Unknown layout kind 'subset'` because `subset` is not registered in the layout factory. Either register or change `subset()` to build a `MaskedGrid`.

### 9.2 `Environment` is freely mutable; mutations silently produce stale caches (HIGH, high)

- `env.bin_centers[0, 0] = 999.0` — silently corrupts KDTree cache.
- `env.connectivity.add_node(999)` — silently invalidates `boundary_bins`, `differential_operator`, `bin_attributes`.
- `env.new_attr = 42` — dataclass allows new attributes.

No version counter, no observer. Either freeze (`@dataclass(frozen=True)`) or detect staleness.

### 9.3 `SubsetLayout.point_to_bin_index` rebuilds KDTree on every call (HIGH, certain)

`environment/transforms.py:613-617`. Repeated `bin_at` queries on a subset env are O(n_queries × n_bins × log n_bins) instead of cached `log n_bins`.

### 9.4 `bin_size` is not uniformly required across factories (MED, certain)

CLAUDE.md says "bin_size is REQUIRED — all Environment creation needs explicit bin_size parameter". False:

- `from_mask` doesn't take `bin_size` (uses `grid_edges`).
- `from_polar_egocentric` splits into `distance_bin_size` and `angle_bin_size`.
- `from_layout` takes `layout_params: dict`.
- `from_nwb` ignores it in scratch-load mode.

### 9.5 Factory `name` argument position varies (MED, certain)

| Factory | Position of `name` |
| --- | --- |
| `from_samples`, `from_polygon`, `from_mask`, `from_layout` | 3rd |
| `from_image` | 4th |
| `from_graph` | 5th |
| `from_polar_egocentric` | 6th |

None enforce keyword-only via `*,`. A user passing positionally hits silent argument confusion.

### 9.6 `from_image` vs `from_mask` names don't telegraph the difference (MED, high)

Both take a 2-D bool array. The real difference is `grid_edges` (mask) vs `bin_size` (image). Names suggest "generic mask vs camera-derived" — a different distinction. Consider `from_grid_mask(active_mask, grid_edges)` and `from_pixel_mask(image_mask, pixel_size)`.

### 9.7 `from_graph` has zero examples in its docstring (LOW, certain)

`environment/factories.py:353-389`. Most complex factory; a user can't construct a graph environment without scrolling to `from_graph`'s call sites.

### 9.8 `mask_for_region(name)` and `region_mask(name)` use different containment predicates (MED, certain)

`environment/regions.py`. `mask_for_region` calls `bins_in_region` which uses `shapely.contains_xy`. `region_mask` calls the `regions_to_mask` free function which uses `covers` or `contains` based on `include_boundary`. For polygon regions where bin centers lie on the boundary, the two can disagree. Either deprecate one or guarantee equivalence.

### 9.9 Region API has FOUR mutation paths (LOW, certain)

`add`, `update_region`, `__setitem__`, `__delitem__`/`remove`, plus full-replace. `add` raises on duplicate; `__setitem__` warns. No idempotent `set`.

### 9.10 `Regions.remove()` silently no-ops on missing key (LOW, certain)

`regions/core.py`. Differs from `dict.pop` (raises) and `set.remove` (raises). Lenient — but loses error info.

### 9.11 `bin_attributes` and `edge_attributes` are eager `cached_property` DataFrames (MED, certain)

`environment/metrics.py:236-252, 299-316`. For 100k bins, accessing `env.bin_attributes` from tab completion silently allocates. Convert to methods to signal cost.

### 9.12 `differential_operator` is O(n³) `cached_property` (MED, certain)

`environment/fields.py:91-103`. Stray access in interactive exploration burns it.

### 9.13 `compute_kernel(cache=True)` and `smooth` silently mutate env state (MED, certain)

CLAUDE.md says `env.clear_cache()` is needed before parallel rendering. The reason is pickle of a kernel cache is large; clearing avoids OOM in workers. Link between symptom and cause is unclear.

### 9.14 `bin_at` and `map_points_to_bins` divergence (HIGH, certain) — already in §6.10

### 9.15 Six "bin_*" methods with confusing input/output orientation (MED, certain)

`bin_at(points)→indices`, `bin_center_of(indices)→coords`, `bin_centers→all coords`, `bin_sequence(times,positions)→indices`, `bin_sizes→areas`, `bin_attributes→DataFrame`. Consider `index_of(points)`, `centers_of(indices)`.
