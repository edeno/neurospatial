**Cross-cutting patterns**

- **Weighted-stat NaN/length handling is the dominant theme.** A single class of bug — weights not validated for length/sign and not filtered alongside NaN angles — recurs across at least 8 findings (numerical-robustness, statistical-validity, silent-failure, ux-ergonomics, tests) all centered on `rayleigh_test` and the weighted descriptive stats near `circular.py:218-237`, `561-607`, `973-985`. Fixing the underlying validation/filtering once resolves most criticals.
- **Convention mismatches:** math vs. compass polar (plotting), `[-π,π]` vs half-open `(-π,π]` (`wrap_angle` annotation + docs), and "Range [0,n]" / "angles will be wrapped" docstrings that don't match behavior.
- **Non-reproducible RNG defaults** (`rng=None`) recur across shuffle/surrogate/util entry points.
- **NumPy-style docstring drift:** non-standard `Properties` section flagged 3×; stale `neurospatial.metrics.phase_precession` module path.

**Recommended order**

1. Fix the weighted-circular validation/NaN-filtering core in `circular.py` (length + sign checks on `weights`, filter weights with angles on NaN removal, handle zero-weight-sum). This single fix clears both silent-failure criticals, the two numerical-robustness criticals, and several important/suggestion items — highest impact per change. Add the missing negative-weights regression test alongside.
2. Fix `compute_shuffle_pvalue` NaN undercounting (important; biases scientific p-values toward false significance) and `shuffle_cell_identity` neuron-count mismatch.
3. Fix the QUICKSTART `circular_basis_metrics` example (important, user-facing TypeError) and the `plot_circular_basis_tuning` convention mismatch.
4. Sweep remaining docs/RNG/annotation suggestions (wrap_angle range, stale module path, `Properties` headers, `rng=None`, surrogate `smoothing_window>=1` validation) last.

## `stats`

### Critical (must fix before use)
- [numerical-robustness / statistical-validity / silent-failure / ux-ergonomics] **Weighted circular functions never validate `len(weights) == len(angles)`; broadcastable/length-1 weights silently produce wrong, out-of-range results** — `src/neurospatial/stats/circular.py:218-237`. `_mean_resultant_length` (and via it `mean_resultant_length`, `circular_variance`, `circular_mean`, `rayleigh_test`) multiplies `weights_norm * cos_angles` with no shape check, so a length-1 weights array silently broadcasts (e.g. R=4.95, outside [0,1]; z=49, n_eff=2); non-broadcastable lengths raise a cryptic broadcast ValueError. Fix: add a shared `_validate_weights(angles, weights)` asserting equal shape (and non-negativity) used by every weighted entry point.
- [numerical-robustness / silent-failure] **`rayleigh_test` drops NaN angles but not the paired weights, scrambling alignment / crashing / inflating `n_eff`** — `src/neurospatial/stats/circular.py:562-594`. `_validate_circular_input` removes NaN angles and returns a shorter array, but `weights` is passed unfiltered to `_mean_resultant_length` (line 569) and summed for `n_eff = sum(weights)` (line 589), so weights mis-pair with surviving angles and include the dropped NaN's weight; usually raises a cryptic `(5,) (4,)` broadcast error, otherwise silently wrong p-value. Fix: have the validator co-filter weights with the same NaN mask and require matching lengths up front; compute `n_eff` from the filtered weights.
- [numerical-robustness] **`rayleigh_test` with all-zero weights raises a bare `ZeroDivisionError` instead of returning NaN/raising clearly** — `src/neurospatial/stats/circular.py:587-607`. `_mean_resultant_length` returns NaN for zero weight-sum, but `rayleigh_test` then hits `(2*z - z**2)/(4*n_eff)` with `n_eff=0.0`, raising an opaque crash (e.g. all-zero spike-count weights from a dead cell). Fix: guard the degenerate case (skip the finite-sample correction when `n_eff <= 0` and raise an actionable error or return NaN/`(0.0,1.0)` consistently).

### Important (should fix)
- [algorithmic-correctness / numerical-robustness / statistical-validity / silent-failure / ux-ergonomics] **`rayleigh_test`: NaN/length/sign weight handling on the documented count-based path** — `src/neurospatial/stats/circular.py:485-594`. Beyond the NaN-misalignment and zero-weight crashes above, the function also never checks `weights >= 0`: a negative weight yields `z = -1.40, p = 1.0` (a Rayleigh z is mathematically `>= 0`), silently masking a caller's buggy weight array. Fix: coerce weights to float64 and reject mismatched length and negative values with actionable messages mirroring `_validate_circular_input`.
- [statistical-validity] **Weighted Rayleigh `n_eff` computed from unfiltered weights gives wrong effective sample size** — `src/neurospatial/stats/circular.py:561-594`. Even when broadcasting does not crash, `z = n_eff * R**2` and the finite-sample correction use `n_eff = sum(weights)` over the original (pre-NaN-removal, unaligned) weights, corrupting the test statistic. Fix: derive `n_eff` only after weights are aligned/filtered to the surviving angles.
- [numerical-robustness / silent-failure] **`compute_shuffle_pvalue` silently undercounts NaN null scores, biasing p-values toward significance** — `src/neurospatial/stats/shuffle.py:1000-1018`. `nan >= observed` and `nan <= observed` are both False, so a failed-shuffle NaN is treated as "not extreme", deflating k and shrinking the p-value with no warning (e.g. returns 0.2 for nulls `[1, nan, 3, 4]`). Fix: detect non-finite null scores and either raise, or warn and reduce n to the count of finite nulls; document the policy.
- [silent-failure] **`shuffle_cell_identity` does not check `spike_counts` neuron count matches `encoding_models` row count** — `src/neurospatial/stats/shuffle.py:306-313`. The cell-identity null reassigns place fields to spike trains, so the two neuron counts must match, but a 3-neuron `spike_counts` against a 5-neuron `encoding_models` runs silently and yields a wrong null (or a crash deep in the decoder). Fix: raise a `ValueError` up front if `spike_counts.shape[1] != encoding_models.shape[0]`.
- [units-and-frames] **`plot_circular_basis_tuning` uses a compass polar convention (0=North, clockwise) that disagrees with the math-convention phase returned by `circular_basis_metrics`** — `src/neurospatial/stats/circular.py:1680-1682`. The plot sets `set_theta_zero_location("N")`/`set_theta_direction(-1)` while `circular_basis_metrics` returns `atan2(beta_sin, beta_cos)` (0=East, CCW), so a reported phase of 0 appears at the top and +π/2 to the right — the visualized peak silently disagrees with the numeric output. Fix: make the plot use the same convention (0 rad at right, CCW) or prominently document the compass convention and add a regression test tying plotted peak to the reported phase.
- [tests] **No regression test for negative weights in weighted circular statistics** — `tests/stats/test_circular_metrics.py:233-285`. Only non-negative and all-zero weights are tested; negative weights silently produce R>1 and false-significant Rayleigh results, an uncovered scientific failure mode. Fix: add a test asserting rejection (or correct values) for negative weights in `rayleigh_test`/`mean_resultant_length`/`circular_mean`/`circular_variance`, paired with source-side validation.
- [api-consistency] **Stale module path: docstring points to nonexistent `neurospatial.metrics.phase_precession`** — `src/neurospatial/stats/circular.py:31-34`. The "Which Function Should I Use?" docstring (and provenance comments at lines 123/128/134) reference a `neurospatial.metrics` package that does not exist; the functions live in `neurospatial.encoding.phase_precession`, so following the cross-reference hits ModuleNotFoundError. Fix: update to `neurospatial.encoding.phase_precession` and refresh the `metrics/` provenance comments.
- [docs-consistency] **QUICKSTART example calls `circular_basis_metrics` with non-existent `var_sin`/`var_cos`/`cov_sin_cos` args (raises TypeError)** — `.claude/QUICKSTART.md:797-802`. The real signature is `circular_basis_metrics(beta_sin, beta_cos, cov_matrix=None)`; the example crashes verbatim. Fix: pass the 2×2 `cov_params()[1:3,1:3]` as `cov_matrix`.

### Suggestions (nice to have)
- [numerical-robustness] **`circular_linear_correlation` returns `(0.0, 1.0)` for all-identical angles only by accident of `max(0.0, nan)`** — `src/neurospatial/stats/circular.py:717-743`. Constant angles make `r_cs` NaN, the `r_cs > 0.9999` guard does not fire, and only Python's `max(0.0, nan)` argument ordering yields the correct degenerate result (fragile; `np.maximum` would emit NaN). Fix: add an explicit constant-angle guard mirroring the `linear_values` guard and warn.
- [numerical-robustness / tests / silent-failure] **`generate_inhomogeneous_poisson_surrogates` does not validate `smoothing_window >= 1` despite the docstring** — `src/neurospatial/stats/surrogates.py:160-163` (also `:142`). `smoothing_window=0`/negative passes straight to scipy and raises a cryptic `RuntimeError: incorrect filter size`. Fix: add `if smoothing_window < 1: raise ValueError(...)` and a test pinning it.
- [statistical-validity] **Rayleigh docstring "Range: [0, n]" is misleading for the weighted (count) convention** — `src/neurospatial/stats/circular.py:513-515`. Under count weighting `z` ranges over `[0, sum(weights)]`, not `[0, len(angles)]`. Fix: clarify the Returns block that for weighted input the range is `[0, sum(weights)]`.
- [units-and-frames] **`rayleigh_test` docstring claims out-of-range angles "will be wrapped" but no wrapping occurs** — `src/neurospatial/stats/circular.py:499-501`. `_validate_circular_input` is called with `check_range=False`, so the wrap branch is dead (harmless since only sin/cos are used). Fix: either state angles are used directly via sin/cos (wrap-invariant) or actually enable wrapping.
- [tests] **`generate_jittered_spikes` window-clipping can collapse spikes onto an identical boundary (untested coincident-spike behavior)** — `tests/stats/test_stats_surrogates.py:282-291`. `np.clip` can produce duplicate (zero-ISI) spike times under tight windows + large jitter, silently feeding artifacts to downstream ISI/burst analysis. Fix: add a tight-window/large-jitter test pinning the intended behavior (allow duplicates, or avoid coincident times).
- [silent-failure] **`generate_jittered_spikes` silently corrupts output when window bounds are reversed** — `src/neurospatial/stats/surrogates.py:328-329`. `window=(1.0, 0.0)` collapses all spikes to the smaller bound with no error. Fix: validate `window[0] <= window[1]` and raise.
- [reproducibility-rng / ux-ergonomics] **Shuffle/surrogate generators default `rng=None` (non-reproducible) — easy footgun for null-distribution work** — `src/neurospatial/stats/shuffle.py:85-90` (and `_utils.py:27-28`). RNG hygiene is otherwise excellent (generator built once, never reseeded in loops, no global state), but a forgotten seed silently yields irreproducible p-values that go into papers. Fix (optional): emit a one-time UserWarning when `rng is None` in inference paths, or prominently cross-reference the seed requirement in QUICKSTART.
- [api-consistency / ux-ergonomics / docs-consistency] **`CircularBasisResult` docstring uses a non-standard "Properties" section header** — `src/neurospatial/stats/circular.py:1131-1135`. numpydoc does not recognize "Properties", so `design_matrix` (the key GLM accessor) will not render. Fix: rename to "Attributes" (or document on the property itself), consistent with `ShuffleTestResult`.
- [ux-ergonomics] **`circular_basis` docstring example uses unseeded `np.random.uniform`, making it non-reproducible** — `src/neurospatial/stats/circular.py:1297-1299`. Inconsistent with the rest of the module (which uses `default_rng(seed)`) and models the exact RNG pattern the shuffle design principles warn against. Fix: use `np.random.default_rng(0).uniform(...)`.
- [ux-ergonomics] **`wrap_angle` type annotation is `NDArray`-only but the function accepts and returns scalars** — `src/neurospatial/stats/circular.py:1085-1106`. `wrap_angle(3*np.pi/2)` returns a Python float; the narrow annotation makes type checkers flag legitimate scalar calls. Fix: broaden to `float | NDArray[np.float64]` and add a scalar docstring example.
- [docs-consistency] **API_REFERENCE describes `wrap_angle` range as closed `[-π, π]` but it is half-open `(-π, π]`** — `.claude/API_REFERENCE.md:564`. `-π` maps to `+π`; the boundary convention matters for code branching at ±π. Fix: change the comment to `(-π, π]`.

### Strengths
- **performance-memory**: nothing flagged — no performance or memory issues surfaced for this module.
- **reproducibility-rng**: RNG handling is well-engineered — the single confirmed item is only a documented-default suggestion. Generators are built once via `_ensure_rng` before loops, never reseeded inside loops, never touch the legacy global `np.random` state, and no set/dict iteration feeds numerical output (verified across all shuffle/surrogate functions); reproducibility-with-seed is unit-tested.
- **statistical-validity**: degenerate-input guards are largely present and correct (explicit guards for constant `linear_values` and near-1 `r_cs`, zero-weight-sum NaN handling); the surviving issues are concentrated in the weighted-Rayleigh path rather than systemic.
