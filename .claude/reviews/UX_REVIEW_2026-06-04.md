# Neurospatial UX and API Review

> **Status: superseded / resolved (2026-07-03).** Historical v0.6 review artifact — its findings were remediated and verified against `main`; retained for provenance. Do not treat its API references as current.

Review date: 2026-06-04

Scope: repository-level UX review focused on beginner to intermediate Python neuroscientists, especially users collecting data from hundreds to thousands of neurons. This review inspected the package structure, public API, docs, examples, scaling paths, NWB I/O, result objects, and error ergonomics. No source files were changed as part of the review.

## Executive Summary

`neurospatial` is a strong scientific Python toolkit for spatial neuroscience. Its core value is turning continuous positions and task geometries into discrete spatial environments, then using those environments for occupancy, rate maps, place fields, behavior segmentation, decoding, simulation, NWB I/O, and visualization.

The primitives are genuinely good. `Environment.from_samples`, `env.occupancy`, `compute_spatial_rate`, `compute_spatial_rates`, `decode_position`, region operations, simulation helpers, and result objects show thoughtful engineering. The main UX problem is not missing algorithms. The main UX problem is that scientists must still assemble too much of the analysis workflow by hand.

For the target user, the package should evolve toward:

- Session-first workflows.
- Labeled population data by default.
- Batch-first APIs that scale naturally from 3 neurons to 3000.
- Clear, stable result objects that can round-trip to NWB, xarray, zarr, and DataFrame summaries.
- A single canonical beginner path in docs and examples.

The radical change I recommend is to keep the existing primitives as the composable core, but add a higher-level product layer that owns data, time, units, neuron identity, environment, derived results, plotting, and export.

## What This Repo Is

The package is named `neurospatial`, version `0.5.0`, and declares optional extras for NWB, JAX, xarray, docs, notebooks, animation, and video. See `pyproject.toml:2`, `pyproject.toml:3`, `pyproject.toml:105`, `pyproject.toml:113`, and `pyproject.toml:120`.

The top-level API intentionally exposes a small public surface: `Environment`, `Region`, `Regions`, `CompositeEnvironment`, `bin_spikes_in_time`, and custom exceptions, with lazy submodule loading. See `src/neurospatial/__init__.py:239` and `src/neurospatial/__init__.py:275`.

- Regions and annotation: `src/neurospatial/regions`, `src/neurospatial/annotation`.

The package is broad: a rough search for top-level definitions found hundreds of functions/classes across the codebase. That breadth is useful, but it also raises cognitive load for users who need one reliable path from "I have spikes and positions" to "I have interpretable population results."

## Target User

The core target user is a neuroscientist who:

- Can use NumPy, pandas, matplotlib, and notebooks.
- May not be confident with package internals, class hierarchies, sparse matrices, graph algorithms, or custom data models.
- Has real files from experiments, often NWB or lab-specific formats.
- Needs to analyze tens, hundreds, or thousands of neurons.
- Needs figures, tables, quality-control checks, and exportable results.
- Needs errors that explain what went wrong in domain language.

For this user, API success means:

- The first analysis works in less than 10 lines.
- Unit IDs are never silently lost.
- Time alignment is visible and hard to misuse.
- Full posterior matrices and giant DataFrames are not accidental defaults.
- Docs teach the recommended path, not an older manual reconstruction.

## What Is Working Well

### The Current Front Door Is Improving

The README has a strong "Your First Place Field" example using `compute_spatial_rate`, simulated positions, simulated spikes, and plotting. See `README.md:175`, `README.md:187`, and `README.md:213`.

The examples index is goal-oriented, with entries such as occupancy, place fields, NWB loading, spatial view cells, head-direction cells, object-vector cells, and PSTHs. See `docs/examples/index.md:10`.

Example 01 is beginner-friendly and conceptual. Example 08 explicitly tells users that most should use `compute_spatial_rate` rather than low-level manual pieces.

### The Core Environment API Has Good Shape

`Environment.from_samples` gives a clear entry point for continuous position data and has useful docstring guidance around units and common pitfalls. See `src/neurospatial/environment/factories.py:116` and `src/neurospatial/environment/factories.py:240`.

`env.bin_at` distinguishes exact geometric containment from nearest-neighbor mapping, which is an important scientific distinction. See `src/neurospatial/environment/queries.py:42`.

`env.occupancy` centralizes time allocation, speed filtering, gap handling, smoothing, and validation. This is the right direction for beginner users because it avoids error-prone manual histograms. See `src/neurospatial/environment/trajectory.py:97`.

### The Encoding API Has the Right Batch Direction

`compute_spatial_rate` is the right single-neuron high-level primitive. See `src/neurospatial/encoding/spatial.py:1564`.

`compute_spatial_rates` is even more important for modern datasets because it shares trajectory/occupancy work across neurons. See `src/neurospatial/encoding/spatial.py:1779`.

`SpatialRatesResult` provides population-level methods, iteration, metrics, plotting, `to_xarray`, and `to_dataframe`. See `src/neurospatial/encoding/spatial.py:761`, `src/neurospatial/encoding/spatial.py:993`, and `src/neurospatial/encoding/spatial.py:1408`.

### Decoding Has Good Scientific Guards

`decode_position` validates encoding model shape and non-finite values, warns about risky model inputs, and returns a structured `DecodingResult`. See `src/neurospatial/decoding/posterior.py:266` and `src/neurospatial/decoding/_result.py:29`.

The docstring explicitly warns about dense posterior memory use. See `src/neurospatial/decoding/posterior.py:371`. This warning is valuable, but the API should go further and make safer memory behavior easy.

### Documentation Testing Exists

There is a doc-snippet CI workflow. See `.github/workflows/test_docs.yml:1` and `.github/workflows/test_docs.yml:60`.

There is also a notebook smoke-test workflow, though it currently executes only a limited notebook surface. See `.github/workflows/test_notebooks.yml:46`.

## Highest Priority UX Problems

### 1. There Is No First-Class Real Session Object

Current user flow is "bring your own orchestration." Users manually pass `env`, `times`, `positions`, `spike_times`, `headings`, `events`, `position_bins`, `spike_counts`, and metadata across modules.

Examples:

- Top-level API exposes no `Session` or `Recording` object. See `src/neurospatial/__init__.py:275`.
- NWB position reading returns separate arrays. See `src/neurospatial/io/nwb/_behavior.py:36`.
- NWB unit reading returns spike trains and unit IDs, but the rest of the workflow does not carry those IDs consistently. See `src/neurospatial/io/nwb/_units.py:25`.
- Simulation has `SimulationSession`, but this is not a general analysis session. See `src/neurospatial/simulation/session.py:17`.

Recommendation:

Add a real user-facing `Session` or `Recording` layer:

```python
import neurospatial as ns

session = ns.load_session("rat01_day03.nwb")

session = session.with_environment(
    ns.Environment.open_field(
        position="position",
        bin_size=2.5,
        units="cm",
        regions="maze_regions.json",
    )
)

rates = session.compute_spatial_rates(min_speed=4.0)
fields = rates.detect_fields()
decoded = session.decode_position(model=rates, dt=0.025, return_posterior="summary")
report = session.metrics(["fields", "decoding", "behavior"])
session.save("rat01_day03_neurospatial.nwb")
```

This should be a friendly layer, not a replacement for low-level functions.

### 2. Unit Identity Is Not Preserved Strongly Enough

For thousands of neurons, positional array order is not a sufficient identity model. Users need `unit_id`, cluster label, brain region, quality metrics, session ID, and inclusion/exclusion status to survive every operation.

Current evidence:

- `read_units` returns `unit_ids`. See `src/neurospatial/io/nwb/_units.py:25`.
- `normalize_spike_times` converts inputs to a list of arrays, one per neuron, with no unit metadata. See `src/neurospatial/encoding/_spikes.py:25`.
- `SpatialRatesResult.to_xarray()` uses integer `neuron` coordinates. See `src/neurospatial/encoding/spatial.py:996` and `src/neurospatial/encoding/spatial.py:1057`.
- Tests lock in integer neuron coordinates for the xarray export. See `tests/encoding/test_spatial_xarray_interop.py:37`.
- `to_dataframe(neuron_ids=...)` can accept identifiers, but the identifiers are optional late-bound metadata rather than part of the result itself. See `src/neurospatial/encoding/spatial.py:1410`.

Recommendation:

Add first-class population containers:

```python
spikes = ns.SpikeTrains(
    trains={unit_id: times for unit_id, times in zip(unit_ids, spike_trains)},
    metadata=unit_table,
    time_unit="s",
)

rates = ns.compute_spatial_rates(env, spikes, times, positions)
rates.unit_ids
rates.sel(unit_id=42)
```

At minimum, add `unit_ids` to:

- `SpikeTrains`
- `SpikeCountMatrix`
- `SpatialRatesResult`
- `PlaceFieldsResult` for population detection
- `DecodingResult`
- NWB read/write helpers
- xarray exports

### 3. xarray Should Become a Data Model, Not Just an Export Veneer

Current xarray support is useful but thin. `SpatialRatesResult.to_xarray()` wraps `(n_neurons, n_bins)` as a `DataArray` with integer coordinates. `DecodingResult.to_xarray()` wraps posterior arrays, but does not expose a richer dataset with MAP estimates, entropy, bin centers, times, and provenance.

Recommendations:

- Return `xr.Dataset` for population results, not only `xr.DataArray`.
- Use dims such as `unit`, `time`, `bin`, and `coord`.
- Include coords for `unit_id`, `time`, `bin_center_x`, `bin_center_y`, and region labels.
- Include data variables for rates, occupancy, spike counts, posterior, MAP bin, MAP position, entropy, and masks.
- Preserve attrs for environment ID/hash, units, bin size, smoothing, speed filter, backend, and software version.
- Make dask-backed arrays possible without forcing eager NumPy materialization.

Example target:

```python
rates.dataset

# dims: unit, bin, coord
# coords: unit_id, bin, coord, bin_center
# data_vars: firing_rate, occupancy, spike_count, spatial_information, sparsity
```

External reference: xarray's data model is built around named dimensions and coordinates, which enables label-based indexing and alignment. See https://docs.xarray.dev/en/stable/user-guide/data-structures.html.

### 4. Dense Decoding Is Risky for Long Recordings

`decode_position` stores posterior as a dense `(n_time_bins, n_bins)` float64 array. See `src/neurospatial/decoding/posterior.py:371` and `src/neurospatial/decoding/_result.py:83`.

For 1 hour at 25 ms bins and 20,000 spatial bins, this is roughly:

```text
144,000 time bins * 20,000 bins * 8 bytes = 23.0 GB
```

That is before copies, plotting, export, or multiple sessions.

Recommendations:

- Add `dtype="float32"` support where scientifically acceptable.
- Add `chunks={"time": ..., "bin": ...}`.
- Add `return_posterior` modes:
  - `"full"`: full posterior, explicit memory cost.
  - `"summary"`: MAP, entropy, confidence, maybe top-k bins.
  - `"map"`: only MAP estimates.
  - `"iterator"`: stream chunks.
- Add a preflight memory estimate and require confirmation or explicit override above a threshold.
- Add dask/xarray backed decoding for large outputs.
- Add NWB/zarr export that can write chunks incrementally.

External reference: xarray plus Dask supports chunked, block-wise processing and workloads that do not fit into memory. See https://docs.xarray.dev/en/stable/user-guide/dask.html.

### 5. Docs Teach Multiple Competing First Paths

The README's first place-field path is good, but the docs still contain stale or manual workflows.

Problems:

- `docs/user-guide/workflows.md` uses placeholder loaders such as `load_position_data`. See `docs/user-guide/workflows.md:25`.
- It hand-rolls occupancy and spike histograms even though `env.occupancy` and `compute_spatial_rate` exist. See `docs/user-guide/workflows.md:43`.
- It references undefined `compute_firing_rate`. See `docs/user-guide/workflows.md:304`.
- `docs/user-guide/signal-processing-primitives.md` imports `neurospatial.primitives`, which does not match current example imports. See `docs/user-guide/signal-processing-primitives.md:45`.
- Several docs use `env.plot(field, ...)` while current examples use `env.plot_field(...)`. See `docs/user-guide/spike-field-primitives.md:279` and `docs/user-guide/rl-primitives.md:173`.
- README and docs still mention `v0.4.0` while `pyproject.toml` says `0.5.0`. See `README.md:85`, `docs/index.md:97`, and `pyproject.toml:3`.

Recommendations:

- Make one canonical beginner path:
  1. Load or simulate positions and spikes.
  2. Build environment.
  3. Compute occupancy and spatial rates with `compute_spatial_rate` or `compute_spatial_rates`.
  4. Detect fields.
  5. Export summary table and plot.
- Rewrite `docs/user-guide/workflows.md` around runnable data.
- Keep manual histogram code only in an "under the hood" section.
- Expand doc-snippet CI coverage to the workflow pages most likely to be copied.
- Execute more notebooks in CI or maintain a rotating smoke-test set.

### 6. Region Semantics Are Inconsistent for Point Regions

Point regions are important in neuroscience because goals, feeders, reward wells, cue points, and landmarks are often points rather than polygons.

Current behavior differs by API:

- `bins_in_region()` maps a point region to the bin containing the point. See `src/neurospatial/environment/regions.py:123`.
- `region_membership()` treats point regions as having no area. See `src/neurospatial/environment/regions.py:207`.
- `regions_to_mask()` says point regions always return all False. See `src/neurospatial/ops/binning.py:404`.
- `region_mask()` special-cases points to match what users usually mean by a target. See `src/neurospatial/environment/regions.py:461`.

Recommendation:

Choose one public rule:

- For target/distance/reward APIs, region names should resolve through `env.region_bins(name)` or `env.bins_in_region(name)`.
- For area-only masks, use explicit names like `env.region_area_mask(name)`.
- Point regions should never silently produce all-infinite distances or empty outputs in target-oriented APIs.

Add tests for point regions across:

- `bins_in_region`
- `region_mask`
- `region_membership`
- `distance_to`
- reward fields
- behavior segmentation

### 7. Public Errors Need Beginner-Level Consistency

The repo has custom exceptions, which is good. See `src/neurospatial/_exceptions.py:37` and `src/neurospatial/_exceptions.py:65`.

But some public paths still expose lower-level or confusing errors:

- Direct `Environment()` raises `ValueError("layout parameter is required")`. See `src/neurospatial/environment/core.py:304` and `src/neurospatial/environment/core.py:335`.
- Missing region in `distance_to` raises a plain `KeyError`. See `src/neurospatial/environment/queries.py:528`.
- One graph validation error message references `CLAUDE.md`, which is not user-facing package documentation. See `src/neurospatial/environment/core.py:867`.
- `align_spikes_to_events` documents Inf validation for event times but checks Inf only for spike times. See `src/neurospatial/events/alignment.py:99`, `src/neurospatial/events/alignment.py:141`, and `src/neurospatial/events/alignment.py:156`.
- Some behavior functions inspect `.ndim` before normalizing list-like inputs. See `src/neurospatial/behavior/trajectory.py:132`, `src/neurospatial/behavior/trajectory.py:289`, and `src/neurospatial/behavior/trajectory.py:591`.

Recommendations:

- Normalize array-like inputs at public boundaries.
- Use custom exceptions consistently while preserving stdlib bases.
- Include the parameter name, received shape/type/value, expected shape/type, and a fix.
- Add ergonomic tests for list-like inputs, scalar inputs, negative bin indices, missing regions, Inf event times, and "no internal docs path in public error."

External reference: the Scientific Python Development Guide recommends useful error messages and separating friendly layers from stricter internal layers. See https://learn.scientific-python.org/development/principles/design/.

### 8. Environment and Maze Definition Needs Presets

The environment constructors are powerful, but common tasks still require too much setup.

Examples:

- `Environment.from_samples` has many options. See `src/neurospatial/environment/factories.py:116`.
- `Environment.from_graph` requires graph structure, edge order, spacing, and bin size. See `src/neurospatial/environment/factories.py:326`.
- README plus-maze setup is verbose. See `README.md:336`.
- Simulation has many maze constructors, but these are not exposed as the primary environment-definition UX.

Recommendations:

Add public presets:

```python
env = ns.Environment.open_field(width=100, height=100, bin_size=2.5, units="cm")
env = ns.Environment.track("linear", length=200, bin_size=2.5)
env = ns.Environment.maze("plus", arm_length=80, arm_width=10, bin_size=2.5)
env = ns.Environment.maze("w_track", bin_size=2.5)
```

These presets should sit on top of existing layout/factory machinery.

### 9. Behavior Workflows Require Too Much Pre-Binning

Many behavior and navigation APIs require users to precompute `position_bins`. That is efficient internally, but it is friction for users.

Examples:

- `detect_region_crossings(position_bins, times, ..., env)`: `src/neurospatial/behavior/segmentation.py:321`.
- `detect_laps(position_bins, times, env, ...)`: `src/neurospatial/behavior/segmentation.py:989`.
- `segment_trials(position_bins, times, env, ...)`: `src/neurospatial/behavior/segmentation.py:1620`.

Recommendations:

- Keep low-level binned APIs.
- Add friendly wrappers that accept positions and cache bins:

```python
runs = env.detect_runs(positions=positions, times=times, source="home", target="reward")
laps = env.detect_laps(positions=positions, times=times)
trials = session.segment_trials(start_region="home", end_regions=["left", "right"])
```

### 10. Result Exports Are Inconsistent for Large Populations

Population spatial results can expand to one row per `(neuron, bin)`, which can become huge. See `src/neurospatial/encoding/_base.py:324`, `src/neurospatial/encoding/_base.py:373`, and `src/neurospatial/encoding/_base.py:386`.

Other batch result types export one row per neuron. See `src/neurospatial/encoding/view.py:878`, `src/neurospatial/encoding/egocentric.py:983`, and `src/neurospatial/encoding/directional.py:1391`.

Recommendations:

Use explicit export modes:

```python
rates.to_dataframe(kind="summary")  # default, one row per unit
rates.to_dataframe(kind="bins")     # explicit, one row per unit-bin
rates.to_xarray()                   # labeled dense/chunked array data
rates.to_zarr(path)
rates.to_nwb(nwbfile)
```

Default population DataFrame exports should be summaries, not dense bin tables.

## Recommended API Architecture

### Keep Three Layers

1. Friendly workflow layer:
   - `Session`
   - `load_session`
   - `session.compute_spatial_rates`
   - `session.decode_position`
   - `session.metrics`
   - `session.save`

2. Labeled data/result layer:
   - `PositionTrack`
   - `SpikeTrains`
   - `Epochs`
   - `SpatialRatesDataset`
   - `PlaceFieldsTable`
   - `DecodedPositionDataset`

3. Cranky primitive layer:
   - Current functions such as `compute_spatial_rate`, `compute_spatial_rates`, `bin_spikes_in_time`, `decode_position`, `env.occupancy`, `env.transitions`.
   - These should stay testable, array-friendly, and composable.

This matches Scientific Python design guidance: keep scientific logic in functions over standard data types, but allow friendly layers to provide domain-specific defaults.

### Session Should Be Mostly Immutable

Avoid a mutable object where tab completion shows methods that are invalid until hidden state is set. Prefer methods that return a new session or result:

```python
raw = ns.load_session("file.nwb")
with_env = raw.with_environment(...)
aligned = with_env.align(dt=0.025)
rates = aligned.compute_spatial_rates(...)
```

This follows the Scientific Python recommendation to avoid implicit changing state where possible.

### Data Containers Should Preserve Identity

```python
@dataclass(frozen=True)
class SpikeTrains:
    trains: Mapping[str | int, np.ndarray]
    unit_table: pd.DataFrame | None = None
    time_unit: str = "s"
    time_support: Epochs | None = None
```

```python
@dataclass(frozen=True)
class PositionTrack:
    times: np.ndarray
    positions: np.ndarray
    units: str
    columns: tuple[str, ...] = ("x", "y")
```

These can interoperate with NumPy arrays but provide richer defaults for users.

### Decoding Should Use fit/predict

Current decoding expects pre-binned spike counts and encoding models. See `src/neurospatial/decoding/posterior.py:266` and `src/neurospatial/decoding/_binning.py:24`.

Recommended user-facing layer:

```python
decoder = ns.PositionDecoder(dt=0.025, model="poisson")
decoder = decoder.fit(session, rates=rates, train="first_half")
decoded = decoder.predict(session, return_posterior="summary")
score = decoder.score(session, metric="median_error")
```

External reference: scikit-learn's estimator interface gives users a familiar `fit`, `predict`, `predict_proba`, and `score` model. See https://scikit-learn.org/stable/developers/develop.html.

### NWB Should Be a Round-Trip Story

PyNWB is the Python reference API for NWB and supports large-file workflows through lazy loading, iterative writes, and compression. See https://nwb.org/tools/core/pynwb/.

Current gaps:

- Position and pose reads are eager. See `src/neurospatial/io/nwb/_behavior.py:85` and `src/neurospatial/io/nwb/_pose.py:84`.
- `write_place_field` supports `(n_bins,)` or `(n_time, n_bins)`, but not a population `unit` axis. See `src/neurospatial/io/nwb/_fields.py:150`.
- Static fields are stored as `(1, n_bins)`. See `src/neurospatial/io/nwb/_fields.py:281`.

Recommendations:

- Add `load_session(..., lazy=True)`.
- Add round-trip support for `SpatialRatesDataset`, `PlaceFieldsTable`, and `DecodedPositionDataset`.
- Preserve unit table links or explicit unit IDs.
- Support chunked writes for large posterior/rate-map arrays.

### Learn From Pynapple's Neuro-Specific UX

Pynapple centers time series, timestamps, timestamp groups, intervals, time support, and restriction to epochs. See https://pynapple.org/ and https://pynapple.org/user_guide/01_introduction_to_pynapple.html.

`neurospatial` does not need to copy Pynapple, but it should take the same lesson: neuroscience users benefit from domain objects for time, intervals, and populations. A `Session` should know about time support, epochs, spike groups, and position tracks.

## Proposed Golden Path

The beginner path should be this exact workflow in README, docs, and examples:

```python
import neurospatial as ns

session = ns.example_session("open_field", n_units=20, duration=600)

session = session.with_environment(
    ns.Environment.from_samples(session.position.values, bin_size=2.5, units="cm")
)

rates = session.compute_spatial_rates(min_speed=4.0, bandwidth=5.0)
fields = rates.detect_fields()

summary = fields.to_dataframe()
summary.head()

rates.plot(unit_id=summary.iloc[0]["unit_id"])
```

Then show the real data version:

```python
session = ns.load_session("rat01_day03.nwb")
session = session.with_environment(kind="open_field", bin_size=2.5, units="cm")
rates = session.compute_spatial_rates(min_speed=4.0)
rates.to_dataframe(kind="summary")
```

Avoid manual occupancy and histogram code in the first path.

## API Design Principles To Apply

1. Friendly APIs should accept domain objects and common arrays.
2. Core functions should remain small, typed, and testable.
3. Population results should preserve unit identity automatically.
4. Dense outputs should be opt-in when they can be huge.
5. Default DataFrame exports should be summaries.
6. xarray/zarr/NWB should carry dense labeled arrays.
7. Errors should name the parameter, received value/shape, expected value/shape, and likely fix.
8. Docs should teach only one recommended path before showing alternatives.
9. Optional backend behavior should be explicit. If JAX or Dask is partial, name exactly which operations stay on device or remain lazy.
10. The package should make the scientifically safe path the easiest path.

## Final Recommendation

Do not radically rewrite the working primitives first. They are valuable and already encode a lot of domain knowledge.

Radically change the user-facing layer:

- Add `Session`.
- Add labeled population containers.
- Make unit identity durable.
- Make large-data safety explicit.
